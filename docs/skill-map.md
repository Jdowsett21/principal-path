# Skill Map

This map translates "I want principal-level engineering ability" into trainable capability areas.

## Level Model

### Level 1: Builder

- can implement features with guidance
- knows frameworks better than fundamentals
- limited systems intuition

### Level 2: Owner

- can independently ship features
- understands trade-offs in a local codebase
- handles moderate debugging and delivery pressure

### Level 3: Senior

- owns production systems end to end
- makes pragmatic trade-offs
- debugs across service and database boundaries
- communicates clearly with product and design

### Level 4: Principal

- designs systems for scale and reliability
- leads through ambiguity
- anticipates failure modes and operational cost
- improves engineering standards and architecture across teams

### Level 5: CTO / Consultant

- aligns technical direction with business constraints
- guides organizations through architecture, scale, risk, and hiring decisions
- communicates clearly with executives and engineers

## Competency Areas

## 1. Computer Science Foundations

Needs:

- time and space complexity
- core data structures
- algorithms intuition
- concurrency basics
- networking fundamentals
- operating systems basics

What principal-level looks like:

- uses fundamentals to make architecture and performance decisions
- recognizes hidden complexity costs early

## 2. Databases and Data Modeling

Needs:

- relational modeling
- indexing
- transactions and isolation levels
- query planning
- replication
- partitioning
- eventual consistency trade-offs

What principal-level looks like:

- chooses data models based on scale, correctness, and change patterns
- can reason about operational bottlenecks before they happen

## 3. Distributed Systems

Needs:

- consistency vs availability
- retries and idempotency
- messaging and queues
- backpressure
- coordination
- failure modes
- multi-region trade-offs

What principal-level looks like:

- can design systems that degrade gracefully under stress
- avoids naive assumptions about network reliability

## 4. Scalability and Performance

Needs:

- caching layers
- load balancing
- horizontal vs vertical scaling
- latency budgets
- throughput analysis
- benchmarking
- bottleneck diagnosis

What principal-level looks like:

- identifies likely scaling limits from architecture alone
- balances performance work against product value

## 5. Reliability and Operations

Needs:

- observability
- logs, metrics, traces
- SLIs and SLOs
- alerting
- incident response
- postmortems
- safe deploy practices

What principal-level looks like:

- designs for operability, not just correctness
- treats on-call experience as a design input

## 6. Architecture and System Design

Needs:

- service boundaries
- modularity
- API design
- event-driven patterns
- monolith vs microservices judgment
- migration planning
- build vs buy decisions

What principal-level looks like:

- explains why a design is appropriate given team size, roadmap, and risk
- chooses boring solutions when boring is the right answer

## 7. Security and Risk

Needs:

- auth and authz
- secrets handling
- common web vulnerabilities
- threat modeling
- auditability
- data privacy

What principal-level looks like:

- sees security and compliance as system properties, not afterthoughts

## 8. Product and Business Judgment

Needs:

- scope trade-offs
- value vs effort decisions
- delivery sequencing
- cost awareness
- customer impact reasoning

What principal-level looks like:

- picks the solution that best fits the business, not the one that sounds smartest

## 9. Leadership and Communication

Needs:

- technical writing
- architecture reviews
- mentoring
- stakeholder communication
- influencing without authority
- roadmap framing

What principal-level looks like:

- makes the whole team better
- reduces confusion and improves decision quality

## Role Signals From The Accessible Job Post

The accessible Indeed role strongly reinforced these areas:

- ownership under ambiguity
- reliability and correctness
- production judgment
- Postgres fluency
- observability
- cloud and container environments
- incident response
- skepticism toward AI-generated code

These should be early priorities in the curriculum.
